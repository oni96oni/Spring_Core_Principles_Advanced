        @RestController = @Controller + @ResponseBody

    로그 추적기 - 요구사항
        모든 PUBLIC 메서드의 호출과 응답 정보를 로그로 출력
        애플리케이션의 흐름을 변경하면 안됨
            로그를 남긴다고 해서 비즈니스 로직의 동작에 영향을 주면 안됨
        메서드 호출에 걸린 시간
        정상 흐름과 예외 흐름 구분
        예외 발생시 예외 정보가 남아야 함
        메서드 호출의 깊이 표현
        HTTP 요청을 구분
            HTTP 요청 단위로 특정 ID를 남겨서 어떤 HTTP 요청에서 시작된 것인지 명확하게 구분이 가능해야함
            트랜잭션 ID (DB 트랜잭션X), 여기서는 하나의 HTTP 요청이 시작해서 끝날 때 까지를 하나의 트랜잭션이라 함

    로그 추적기 V1 - 프로토타입 개발
        테스트는 자동으로 검증하는 과정이 필요! 결과를 따로 콘솔로 볼 필요가 없어야 한다.
        응답값이 없는 경우 자동검증 하려면 여러 테스트 기법이 필요하다.

    로그 추적기 V1 - 적용
        HelloTraceV1을 주입, 스프링 빈으로 등록 당연히 싱글톤

        단순히 두줄의 코드만 추가되는것이 아니라 try - catch 문이 추가 되어서 코드가 지저분해졌다.

        일단 HelloTraceV1 덕분에 직접 로그를 하나하나 남기는것보다는 편하지만 코드가 너무 복잡하다!

    정리
        TraceId를 넘겨주기 위해서 모든 메서드의 파라미터에다가 추가해주는것이 문제!
        만약에 인터페이스도 있다면? 메서드가 몇백개라면? 추가해줘야하는 매개변수와 수정해야하는 코드가 너무 너무 많아진다.
